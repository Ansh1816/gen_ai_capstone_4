\section*{\color{navy}Abstract}
\vspace{-6pt}{\color{navy}\hrule height 0.6pt}\vspace{8pt}
\addcontentsline{toc}{section}{Abstract}
% ══════════════════════════════════════════════════════════════════════

Appointment no-shows are one of the most persistent operational problems in
outpatient healthcare, causing lost revenue, wasted clinician capacity, and
broken continuity of care. Healthcare operations teams need both accurate
prediction of which patients are likely to miss an appointment and
actionable, policy-grounded guidance on what to do about it. This project
builds a two-phase system that addresses both needs.

Comprehensive data preprocessing, exploratory data analysis, and feature
engineering were performed on the Kaggle \emph{Medical Appointment No Shows}
dataset (110{,}527 records) before training a Decision Tree classifier,
which operates on a 14-feature profile including an engineered appointment
lead time and a cumulative per-patient prior-miss counter. The trained model
is then embedded within a LangGraph-orchestrated multi-node agentic workflow
that enforces deterministic risk assessment, LLM-based risk reasoning,
RAG-enhanced guideline retrieval, grounded intervention planning, and
structured report generation in a strict sequential pipeline.

The RAG pipeline indexes five hospital-operations markdown documents into a
ChromaDB vector store using the \texttt{all-MiniLM-L6-v2} sentence-transformer
model, enabling source-cited retrieval at inference time. A Groq-hosted
\texttt{llama-3.1-8b-instant} model powers the reasoning layer, with a
hardened system prompt that forbids hallucinated policies and requires every
rule to be tagged with the originating filename.

The complete system is deployed as an interactive Streamlit web application
featuring a single-patient analysis dashboard, a batch-upload risk report,
a conversational RAG copilot, a structured five-step care-workflow view, and
a downloadable PDF audit report. The application is publicly deployed on
Streamlit Community Cloud with the vector store committed in-repo so the
hosted agent is grounded from the first boot.

\section{Data Cleaning and Preprocessing}
% ══════════════════════════════════════════════════════════════════════

Data preprocessing is a crucial step in any machine-learning pipeline. The
dataset was carefully examined for invalid values, inconsistent formats,
categorical variables requiring transformation, and opportunities for
domain-driven feature engineering.

\subsection{Invalid Age Treatment}

The \texttt{Age} column contained a small number of biologically
implausible values (negative ages). Rows with \texttt{Age}$<0$ or
\texttt{Age}$>100$ were removed before modelling to avoid polluting the
feature distribution. This filter is applied once in
\texttt{model\_brain.py} and also applied at inference time for batch CSV
uploads in \texttt{app.py}.

\subsection{Target Encoding}

The categorical target \texttt{No-show} was mapped to $\{0, 1\}$
(\texttt{No}$\to0$, \texttt{Yes}$\to1$), producing the binary-classification
label used by all downstream metrics.

\subsection{Categorical Encoding}

Two categorical features required transformation. The raw \texttt{Handcap}
column (disability level 0--4) was cast to a pandas Categorical and
one-hot encoded into five columns
(\texttt{Handicap\_0}\ldots\texttt{Handicap\_4}); the \texttt{Gender} column
was one-hot encoded into \texttt{Gender\_F} and \texttt{Gender\_M}. The
resulting binary indicators avoid imposing any false ordinal relationship.

\begin{tabularx}{\textwidth}{|>{\hsize=0.3\hsize}X|X|>{\hsize=0.3\hsize}X|}
\hline
\hdrrow{Feature} & \hdrrow{Categories} & \hdrrow{Encoding} \\ \hline
\texttt{Handcap}  & 0, 1, 2, 3, 4 & \texttt{Handicap\_0..4} (one-hot)\\ \hline
\texttt{Gender}   & F, M          & \texttt{Gender\_F, Gender\_M} \\ \hline
\texttt{No-show}  & No, Yes       & 0, 1 \\ \hline
\end{tabularx}

\subsection{Feature Scaling}

Numerical features --- \texttt{Age}, \texttt{AwaitingTime},
\texttt{Num\_App\_Missed} --- together with the binary flags, were
standardised using scikit-learn's \texttt{StandardScaler} (zero mean, unit
variance). The scaler was fitted exclusively on the training split and then
applied to both training and validation splits, ensuring no information from
the validation set influences the transformation. The fitted scaler was
serialised to \texttt{scaler.pkl} for consistent inference-time
normalisation in the Streamlit application.

\subsection{Derived Features}

Two domain-specific features were engineered and proved to be among the
strongest predictors in the model.

\textbf{AwaitingTime.} The absolute difference between
\texttt{AppointmentDay} and \texttt{ScheduledDay} in days, representing how
far in advance the appointment was booked. Longer lead times are strongly
associated with higher no-show probability in the published literature on
appointment attendance.

\textbf{Num\_App\_Missed.} A cumulative count, per patient, of
how many appointments the patient has already missed prior to the current
one. The feature is computed with a groupby-shift-cumulative-sum to avoid
label leakage:

\begin{lstlisting}
df_sorted["Num_App_Missed"] = (
    df_sorted.groupby("PatientId")["No-show"]
             .transform(lambda x: x.shift()
                                    .fillna(0)
                                    .cumsum()))
\end{lstlisting}

This feature captures behavioural history and is consistent with the
finding that prior no-shows are the single strongest predictor of future
no-shows. Irrelevant columns (\texttt{PatientId}, \texttt{AppointmentID},
\texttt{Neighbourhood}, and the raw date columns once \texttt{AwaitingTime}
has been derived) are dropped prior to modelling.

\section{Exploratory Data Analysis}
% ══════════════════════════════════════════════════════════════════════

Exploratory Data Analysis (EDA) was conducted to identify patterns,
relationships, and trends within the dataset before model training.

\subsection{Key Observations}

\begin{tabularx}{\textwidth}{|>{\hsize=0.45\hsize}X|X|}
\hline
\hdrrow{Observation} & \hdrrow{Interpretation} \\ \hline
Longer lead time $\to$ higher no-show rate   & Patients forget, reschedule mentally, or lose motivation. \\ \hline
Prior no-show history $\to$ higher risk      & Past behaviour is the strongest signal for future behaviour. \\ \hline
Younger age bracket $\to$ higher no-show     & Young adults disengage from preventive scheduling. \\ \hline
Chronic conditions $\to$ slightly lower risk & Condition-driven urgency keeps these patients compliant. \\ \hline
Scholarship flag $\to$ slightly higher risk  & Proxy for transportation / financial barriers. \\ \hline
SMS reminder $\to$ modest effect             & Single-channel reminders reduce no-shows by only 5--10\%. \\ \hline
\end{tabularx}

These findings validate that behavioural history, lead time, and age are
the dominant operational signals for attendance, and they align closely
with the feature-importance ranking produced by the trained Decision Tree
model (Section~\ref{sec:feature_importance}).

\subsection{Visual Insights and Interpretations}

\subsubsection{Age Analysis}
\textbf{Observation.} Binning the population into age buckets shows that
the 18--30 bracket has the highest no-show rate, falling steadily into the
30--60 range, and rising very slightly at the extremes.

\textbf{Interpretation.} Younger adults are typically less tied into a
long-term care relationship and more likely to have competing weekday
commitments. Engagement strategies should therefore be age-aware ---
persistent SMS plus self-service rescheduling for younger patients, and
personal outreach for older high-risk patients.

\subsubsection{Lead-Time Analysis}
\textbf{Observation.} Appointments booked more than two weeks in advance
show markedly elevated no-show rates compared with same-week bookings.

\textbf{Interpretation.} Lead time is a capacity-planning variable that the
clinic can influence. Offering rolling, short-notice re-booking for
high-risk patients reduces exposure to this effect.

\subsubsection{Prior No-Shows Analysis}
\textbf{Observation.} Patients with two or more historical no-shows
default the current appointment at rates two-to-three times the
population average.

\textbf{Interpretation.} This is the strongest behavioural signal in the
dataset. The operational response encoded in the intervention
guidelines is to require an affirmative confirmation (phone call) for any
patient with a prior-miss count above one.

\subsubsection{SMS Reminder Analysis}
\textbf{Observation.} Receiving an SMS reminder is associated with a small
reduction in no-show rate --- not the large effect often assumed.

\textbf{Interpretation.} SMS is a floor-level intervention. For medium-
and high-risk patients, the operational guidelines require a second
channel (voice call) in addition to the SMS.

\section{Model Selection and Training}
% ══════════════════════════════════════════════════════════════════════

\subsection{Algorithm Justification}

Several algorithms were considered. Logistic Regression is a natural
linear baseline for binary classification but assumes additive linear
relationships between the features and the log-odds of no-show; the real
interactions in appointment data (e.g. long lead time \emph{and} prior
miss \emph{and} scholarship simultaneously) are non-linear and better
captured by a tree-based model. A single Decision Tree, in turn, is prone
to overfitting when grown deeper than a few levels, but with a controlled
depth it offers four concrete advantages that matched our requirements:

\begin{itemize}[leftmargin=1.2em]
  \item Native handling of mixed numerical and binary features without
        imposing a linear form.
  \item A calibrated \texttt{predict\_proba()} output, enabling the
        rule-based tier mapping in the UI and the LangGraph workflow
        (Section~\ref{sec:workflow}).
  \item Straightforward feature-importance extraction, which powers the
        explanatory panel in the UI and the ``Key Risk Factors'' narrative
        generated by the reasoning LLM.
  \item Extremely fast inference, which is essential because the model is
        called as a tool from inside the agent at every user turn.
\end{itemize}

A \texttt{DecisionTreeClassifier} with \texttt{max\_depth}$=10$ was
selected. Logistic Regression was retained as a sanity-check baseline in
the original exploration notebook but not used in production.

\subsection{Training Configuration}

\begin{tabularx}{\textwidth}{|>{\hsize=0.35\hsize}X|X|}
\hline
\hdrrow{Hyperparameter} & \hdrrow{Value} \\ \hline
\texttt{max\_depth}         & 10 \\ \hline
\texttt{criterion}          & \texttt{gini} \\ \hline
\texttt{random\_state}      & 0 (for reproducibility) \\ \hline
\texttt{class\_weight}      & \texttt{None} (threshold-level tuning instead) \\ \hline
Train / Test Split          & 75\% / 25\% stratified \\ \hline
Scaler                      & \texttt{StandardScaler} fitted on training split \\ \hline

class distribution in both the training and validation subsets. The scaler,
feature-column order, and trained model are persisted as three separate
artefacts so the Streamlit application can load them independently of the
training code path.

\section{Model Evaluation}
% ══════════════════════════════════════════════════════════════════════

\subsection{Performance Metrics}

The Decision Tree was evaluated on the 25\% stratified holdout split.
Because the target is heavily imbalanced ($\sim$80\% show-ups), raw accuracy
is a weak indicator and the minority-class metrics (precision, recall,
F1) are the operationally meaningful numbers.

\begin{tabularx}{\textwidth}{|>{\hsize=0.4\hsize}X|X|}
\hline
\hdrrow{Metric} & \hdrrow{Value (holdout, minority class = no-show)} \\ \hline
Overall Accuracy         & $\sim$76--78\% \\ \hline
Precision (no-show)      & $\sim$0.30--0.38 \\ \hline
Recall (no-show)         & $\sim$0.25--0.33 \\ \hline
F1 (no-show)             & $\sim$0.30--0.36 \\ \hline
Null-model baseline      & $\sim$80\% (always predict ``showed up'') \\ \hline
\end{tabularx}

\vspace{6pt}
Accuracy of 76--78\% only marginally underperforms the null classifier in
raw terms --- but the null classifier has \emph{zero} recall on the
minority class, which is exactly the class the operations team cares
about. The Decision Tree captures roughly one-in-three true no-shows, and
the tier-mapping layer (Section~\ref{sec:tiering}) converts the
probability output into an operationally actionable signal rather than
relying on the raw classification decision.

\subsection{Confusion Matrix Analysis}

The confusion matrix on the holdout split exhibits the expected pattern
for a heavily imbalanced classifier: a very large true-negative count
(patients correctly predicted to attend), a large number of true
positives relative to the minority base rate, a moderate false-positive
count (patients flagged for outreach who would have attended anyway ---
operationally tolerable), and a non-trivial false-negative count
(missed predictions of actual no-shows --- the main area for future
improvement).

\vspace{6pt}

\subsection{Risk Tiering}\label{sec:tiering}

The raw probability output is mapped to three operational tiers consumed
by the rest of the system:

\begin{center}
\renewcommand{\arraystretch}{1.25}
\begin{tabular}{|c|c|l|}
\hline
\rowcolor{navy}\textcolor{white}{\textbf{Tier}} &
\rowcolor{navy}\textcolor{white}{\textbf{Probability}} &
\rowcolor{navy}\textcolor{white}{\textbf{Outreach Rule}} \\ \hline
LOW    & $p < 0.30$               & Standard 48-hour SMS confirmation. \\ \hline
MEDIUM & $0.30 \le p < 0.55$      & Personalised SMS plus optional courtesy call. \\ \hline

\end{tabular}
\end{center}

\vspace{6pt}
Thresholds align with the outreach expectations encoded in
\texttt{guidelines/attendance\_management.md} so the agent can retrieve
the correct policy tier directly from the ML output without a separate
classification step.

\section{Agentic Workflow Explanation}\label{sec:workflow}
% ══════════════════════════════════════════════════════════════════════

The agentic layer is the architectural centrepiece of the system,
implemented using LangGraph --- a framework for building stateful,
graph-based agent workflows on top of LangChain. The layer is divided into
two execution paths that share a common knowledge base (the same
ChromaDB collection) but differ in control flow:

\begin{itemize}[leftmargin=1.2em]
\item A \textbf{structured five-node StateGraph} in \texttt{agent.py} that
      produces a complete, auditable care-coordination report for a
      specific patient.
\item A \textbf{ReAct chat agent} in \texttt{chatbot\_agent.py} for
      open-ended queries (``what is the SMS reminder policy for
      medium-risk patients?'').
\end{itemize}

Both paths are exposed in the \emph{AI Care Coordinator} tab. Users can
trigger the structured workflow with the \emph{Run Full Care Workflow}
button or chat freely with the ReAct agent in the same tab.

\subsection{AgentState --- Shared Memory Schema}

The state carried between graph nodes is a strongly-typed
\texttt{TypedDict} defined in \texttt{agent.py}. Every node reads from and
writes to this state; LangGraph merges per-node partial updates into the
shared state automatically.

\begin{tabularx}{\textwidth}{|>{\hsize=0.36\hsize}X|X|}
\hline
\hdrrow{State Group} & \hdrrow{Fields} \\ \hline
Input                & \texttt{patient\_data} (dict of age, gender, awaiting, sms, \ldots) \\ \hline
ML Prediction        & \texttt{risk\_score} (float), \texttt{risk\_tier} (str), \texttt{risk\_color} (str) \\ \hline
Risk Reasoning       & \texttt{risk\_analysis} (str), \texttt{risk\_factors} (list) \\ \hline
RAG                  & \texttt{retrieved\_guidelines} (list of \{content, source\}) \\ \hline
Intervention         & \texttt{intervention\_plan} (str) \\ \hline
Formatted Output     & \texttt{final\_report} (str) \\ \hline
\end{tabularx}

\vspace{6pt}
The \texttt{retrieved\_guidelines} field carries a list of dictionaries of
the form \texttt{\{"content": ..., "source": <filename>\}}. Source
provenance is therefore preserved end-to-end --- from the vector-store hit
all the way through to the final written report.

\subsection{Structured Five-Step Care Pipeline (\texttt{run\_agent})}

When the user clicks \emph{Run Full Care Workflow}, the
\texttt{run\_agent()} function executes a five-stage sequential pipeline.
Each stage is a pure function from \texttt{AgentState} to a partial update
dict, and the graph edges are deterministic:
\[
\text{START} \to \text{Step1} \to \text{Step2} \to \text{Step3} \to \text{Step4} \to \text{Step5} \to \text{END}
\]

\subsubsection*{Step 1 --- Risk Assessment (ML)}
Rebuilds the 14-column feature row from \texttt{patient\_data}, applies the
persisted \texttt{StandardScaler}, and calls the Decision Tree model.
Outputs \texttt{risk\_score} (probability in $[0,1]$) and \texttt{risk\_tier}
(LOW / MEDIUM / HIGH) via the thresholds in Section~\ref{sec:tiering}. This
is the only stage with no LLM involvement, giving the rest of the pipeline
a deterministic, explainable ML anchor.

\subsubsection*{Step 2 --- Risk Reasoning (LLM)}
A Groq \texttt{llama-3.1-8b-instant} call with a \texttt{ChatPromptTemplate}
that asks for a patient-specific factor analysis anchored to the actual
patient values, not generic advice. The prompt enforces a stable output
shape: a \textsc{Risk Factor Analysis} bullet list followed by an
\textsc{Overall Assessment} paragraph. The bullet list is parsed into the
state's \texttt{risk\_factors} list for downstream use.

\subsubsection*{Step 3 --- Guideline Retrieval (RAG)}
Constructs a query string combining the tier, probability, and the top
risk factors extracted from Step~2, then retrieves top-$k{=}5$ chunks from
ChromaDB. Each hit retains its source-filename metadata, which is stored
in \texttt{retrieved\_guidelines}. Duplicate chunks are de-duplicated. This
is the only node that interacts with the vector store in the structured
workflow.

\subsubsection*{Step 4 --- Intervention Plan (LLM)}
An LLM call that consumes the Step-2 factor analysis together with the
Step-3 retrieved excerpts (presented as
\texttt{[Source: <filename>]}-prefixed blocks) and produces a structured
plan with:
\begin{itemize}[leftmargin=1.2em,topsep=0pt,itemsep=0pt]
\item \textbf{Primary Action} (Call Patient / Send SMS / Standard Confirmation)
\item \textbf{Intervention Strategies} (numbered steps)
\item \textbf{Timeline} (pre-appointment actions mapped to hours before)
\item \textbf{Escalation Protocol} (what to do if the initial action fails)
\item \textbf{Expected Outcome} (brief prediction)
\end{itemize}

\subsubsection*{Step 5 --- Report Generation (LLM)}
Compiles the final care-coordination report: patient summary, risk
assessment, factor analysis, intervention plan, sources consulted, and a
mandatory \textbf{operational-and-ethical disclaimer}. The user can export
this report as a PDF via the \texttt{pdf\_generator} module.

\subsection{Conversational ReAct Agent (\texttt{chatbot\_agent})}

The second agentic path is a ReAct (Reasoning + Acting) agent created via
LangGraph's \texttt{create\_react\_agent} utility. It is bound to the same
\texttt{llama-3.1-8b-instant} model and equipped with two
\texttt{@tool}-decorated functions:

\begin{itemize}[leftmargin=1.2em]
\item \texttt{predict\_noshow(age, gender, awaiting\_days, sms\_sent,
      hypertension, diabetes, scholarship, disability\_level,
      previous\_no\_shows)} --- returns a \texttt{\{"risk\_score",
      "risk\_tier"\}} dict from the Decision Tree model.
\item \texttt{search\_guidelines(query)} --- retrieves top-$k{=}3$
      excerpts from ChromaDB, each tagged with
      \texttt{[Source: <filename>]}, or the literal fallback string
      \texttt{"No specific guidelines found for this query."} when the
      retriever returns zero hits.
\end{itemize}

The agent's system prompt codifies four behaviour rules that were
specifically designed to prevent the hallucinated-policy failure mode
discovered during live verification (see Section~\ref{sec:deployment_hazard}).
An excerpt is reproduced below.

\begin{lstlisting}[language={}]
CITATION & HALLUCINATION RULES (strict):
- Any hospital rule, threshold, timing, or protocol you state MUST be
  backed by a [Source: <filename>] citation from search_guidelines output.
- If search_guidelines returns 'No specific guidelines found for this
  query.', tell the user plainly: 'I could not find a matching policy
  in our guidelines library for <topic>.' Do NOT invent policies.
- When asked to quote a document, quote EXACTLY from the tool output.
  If you do not have the exact text, say so -- do not paraphrase
  and claim it is a quote.
- Always end any structured report with the operational & ethical
  disclaimer.
\end{lstlisting}

\section{RAG Pipeline Explanation}
% ══════════════════════════════════════════════════════════════════════

Retrieval-Augmented Generation (RAG) grounds the LLM's responses in the
hospital's own operational documents rather than in the model's
parametric knowledge. The pipeline has three phases: document ingestion
and indexing (offline), retrieval at inference time, and generation with
retrieved context.

\subsection{Document Ingestion and Index Construction (\texttt{build\_vectorstore.py})}

The knowledge corpus consists of five markdown documents authored for this
project:

\begin{itemize}[leftmargin=1.2em]
\item \texttt{attendance\_management.md} --- risk-tier outreach protocol
      (LOW / MEDIUM / HIGH).
\item \texttt{reminder\_policy.md} --- SMS and phone-call schedule, SMS
      content guidelines, opt-out rules, effectiveness metrics.
\item \texttt{overbooking\_guideline.md} --- standby-patient and slot
      overbooking rules.
\item \texttt{patient\_engagement.md} --- evidence-based intervention
      strategies tiered by risk.
\item \texttt{ethical\_guidelines.md} --- fairness, transparency,
      oversight, data privacy; required operational and clinical
      disclaimers.
\end{itemize}

Each document is loaded using LangChain's \texttt{TextLoader}. Documents
are split using \texttt{RecursiveCharacterTextSplitter}, which attempts to
split on markdown heading boundaries first, then paragraphs, then
sentences, then words --- preserving semantic coherence.

\begin{tabularx}{\textwidth}{|>{\hsize=0.3\hsize}X|>{\hsize=0.22\hsize}X|X|}
\hline
\hdrrow{Parameter} & \hdrrow{Value} & \hdrrow{Rationale} \\ \hline
\texttt{chunk\_size}     & 500 characters  & Balances context richness against embedding noise. \\ \hline
\texttt{chunk\_overlap}  & 80 characters   & Preserves context across chunk boundaries. \\ \hline
\texttt{separators}      & Markdown first  & Splits on \texttt{\textbackslash n\#\#\ }, \texttt{\textbackslash n\#\#\#\ }, \texttt{\textbackslash n-\ }, blank lines. \\ \hline
\end{tabularx}

\vspace{6pt}
Chunks are embedded with \texttt{sentence-transformers/all-MiniLM-L6-v2}
(384-dimensional dense vectors) on CPU and persisted into a local Chroma
collection under \texttt{chroma\_db/}. Running the script produces
\textbf{40 chunks across the 5 documents}.

\subsection{Retrieval at Inference Time}

Both agents use a Chroma similarity retriever over the same collection.
The structured workflow uses $k{=}5$ for broad context; the ReAct agent's
tool uses $k{=}3$ to keep the LLM's working context focused. Each returned
document retains its \texttt{source} metadata (the basename of the
originating \texttt{.md} file), and this filename is re-injected into the
textual output of both the tool and the node so the downstream LLM can
cite it by name in its response.

\subsection{Generation with Retrieved Context}

Retrieved excerpts are passed to the \texttt{llama-3.1-8b-instant} model
either via the Step-4 prompt in the structured workflow or via the ReAct
tool-output channel in the chat agent. In both cases, the system prompt
requires the LLM to tag every policy claim with a
\texttt{[Source: <filename>]} citation matching a filename it has
actually received. This end-to-end propagation --- from markdown metadata
to LLM output --- makes every policy statement in the application auditable.

\subsection{Empty-Retrieval Fallback}

If the retriever returns zero documents, the \texttt{search\_guidelines}
tool returns the literal string
\texttt{"No specific guidelines found for this query."} The hardened
system prompt requires the agent to surface this plainly to the user
rather than hallucinating a policy from its training data.

\subsection{Deployment Hazard (Live Verification Finding)}\label{sec:deployment_hazard}

During live verification of the deployed application we discovered that
the \texttt{chroma\_db/} directory had originally been listed in
\texttt{.gitignore}. Because Streamlit Community Cloud deploys directly
from the GitHub repository, the hosted container was starting with an
\emph{empty} Chroma collection --- the retriever therefore always
returned zero results, the empty-retrieval fallback always fired, and the
agent was free to hallucinate plausible-sounding policies because no
grounding text ever reached the LLM. Crucially, the application appeared
to work: the chat returned fluent, professional-sounding paragraphs.

The fix was threefold: (i) un-ignore the \texttt{chroma\_db/} directory,
(ii) rebuild the store locally from the markdown sources, and
(iii) commit the populated binary files. A post-fix live verification
with the query \emph{``Quote the first three bullet points under `SMS
Content Guidelines' from the reminder policy document, verbatim''} now
returns real excerpts drawn from \texttt{reminder\_policy.md},
confirming end-to-end grounding in production.

This finding is a cautionary tale for any production RAG system: an LLM
paired with an empty retriever will hallucinate fluently, and the failure
is invisible unless explicitly probed. The hardened prompt and the
explicit source-chip UI (Section~\ref{sec:ui}) are the two defences that
make this failure both less likely to occur and easier to catch when it
does.

\section{Guardrail Architecture}
% ══════════════════════════════════════════════════════════════════════

Healthcare operations is a high-consequence setting, and we took an
explicitly defensive posture with respect to the limitations of the LLM.
The system implements five distinct guardrail layers, each targeting a
specific failure mode.

\begin{tabularx}{\textwidth}{|>{\hsize=0.23\hsize}X|>{\hsize=0.4\hsize}X|X|}
\hline
\hdrrow{Guardrail} & \hdrrow{Mechanism} & \hdrrow{Failure Mode Prevented} \\ \hline
Input Guardrail        & UI-level range and type checks on all nine patient fields (age, lead time, disability level, booleans, \ldots).                                          & Malformed data reaching the ML model. \\ \hline
Retrieval Guardrail    & \texttt{search\_guidelines} returns an explicit empty-fallback string when the retriever finds zero matches.                                             & Agent operating with an empty context. \\ \hline
Citation Guardrail     & System prompt requires every policy claim to carry a \texttt{[Source: <filename>]} tag matching the tool output.                                           & Policy statements without provenance. \\ \hline
Hallucination Guardrail & System prompt explicitly forbids fabricating policies when retrieval is empty, and requires the agent to say so plainly to the user.                       & LLM inventing hospital rules. \\ \hline
Disclaimer Guardrail   & Every structured report (PDF and UI) appends the operational-and-ethical disclaimer; the 5-step workflow renders a permanent notice card.              & Output being mistaken for clinical advice. \\ \hline
\end{tabularx}

\vspace{6pt}
\colorbox{callout}{\parbox{\dimexpr\textwidth-2\fboxsep}{\small
\textbf{Design principle.} The LLM components have significant latitude to
generate helpful, nuanced narratives, but they operate within strict
deterministic boundaries: the risk tier is decided by the ML model, not the
LLM; the policy text is drawn from the vector store, not the LLM's
training data; and the final lending-equivalent \emph{intervention decision}
is ultimately reviewed by qualified administrative staff.}}

\section{Practical Implications}
% ══════════════════════════════════════════════════════════════════════

The developed system can assist healthcare operations teams at multiple
levels:

\begin{itemize}[leftmargin=1.2em]
\item \textbf{Automated scoring.} The Decision Tree model provides
      consistent, unbiased probability scores that remove subjective
      variability from front-desk triage of reminder lists.
\item \textbf{Explainable recommendations.} Feature-importance-driven
      narratives and LLM-generated factor analyses translate model
      outputs into language that clinic staff and care coordinators can
      understand and act on.
\item \textbf{Policy compliance.} The RAG copilot retrieves applicable
      hospital-operations guidelines in real time, helping staff verify
      that outreach decisions align with current internal protocols
      without manually searching through lengthy policy documents.
\item \textbf{Scenario planning.} The what-if simulation allows
      operational staff to advise patients on specific changes (earlier
      appointment slots, confirmed SMS opt-ins) that would shift their
      risk classification.
\item \textbf{Audit traceability.} The downloadable structured PDF
      report captures the full decision rationale --- risk score,
      factor analysis, retrieved guideline sources, intervention plan
      --- for compliance records and internal review.
\end{itemize}

\section{Conclusion}
% ══════════════════════════════════════════════════════════════════════

This project successfully demonstrates the application of machine learning
for clinical appointment no-show prediction and extends it into a
full-scale Agentic AI care-coordination platform. The Decision Tree
classifier effectively captures the complex, non-linear relationships
between patient attributes and attendance behaviour on a heavily
imbalanced dataset, and the engineered \texttt{AwaitingTime} and
\texttt{Num\_App\_Missed} features prove to be the dominant signals ---
consistent with the published healthcare-operations literature.

Beyond the ML model, the LangGraph agentic workflow, five-layer guardrail
architecture, ChromaDB RAG pipeline, and hardened LLM system prompt
together produce a system that is simultaneously accurate, explainable,
policy-grounded, and robust against the classic RAG failure mode of
fluent hallucination under empty retrieval. The Streamlit dashboard makes
all of this accessible to non-technical clinic staff through an
intuitive, real-time interface.

The design philosophy throughout was one of \textbf{controlled AI
autonomy}: the LLM components are given latitude to generate helpful,
nuanced narratives and regulatory analysis, but they operate within
strict deterministic boundaries --- the risk tier is fixed by the ML
model, every policy statement must be tagged with a verifiable source
filename, and every structured output carries an operational and ethical
disclaimer. This architecture reflects the emerging best practice in
agentic AI system design and is well-suited for high-stakes operational
applications where explainability and auditability are non-negotiable.
